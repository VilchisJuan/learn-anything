**Stack memory** is a region of a computer's memory that stores temporary data created during the execution of a program. It operates on a **last-in, first-out (LIFO)** principle — the most recently added item is always the first to be removed.

The stack is managed automatically by the CPU and the operating system. You do not need to explicitly allocate or free stack memory; it happens as functions are called and return.

## How the Stack Works

Every time a function is called, the program creates a **stack frame** (also called an activation record) and pushes it onto the top of the stack. This frame holds:

- The function's **local variables**
- The **return address** — where execution should resume after the function returns
- **Function parameters** passed by value
- Saved **register values**

When the function returns, its stack frame is popped off and that memory is immediately reclaimed — no cleanup code required.

```c
void greet(char* name) {
  char greeting[50]; // allocated on the stack automatically
  sprintf(greeting, "Hello, %s", name);
} // frame is destroyed here, memory reclaimed instantly

int main() {
  greet("Alice"); // pushes a frame onto the stack
  greet("Bob");   // pushes another frame, first is already gone
  return 0;
}
```

## The Stack Pointer

The CPU keeps a **stack pointer** — a register that always points to the top of the current stack. Pushing a frame decrements the pointer (stacks typically grow downward in memory), and popping a frame increments it. This is why stack allocation is extremely fast: it is just arithmetic on a single register.

## Characteristics

| Property | Value |
|---|---|
| Allocation | Automatic — managed by the CPU |
| Speed | Very fast — just moving a pointer |
| Size | Limited (typically 1–8 MB per thread) |
| Lifetime | Tied to the enclosing function's scope |
| Thread safety | Each thread has its own private stack |

## Stack Overflow

A **stack overflow** occurs when the stack grows beyond its allocated size. The most common causes are:

- **Infinite recursion** — a function calls itself without a base case, pushing frames endlessly
- **Very large local variables** — declaring a massive array on the stack exhausts space quickly

```c
// This will crash with a stack overflow
void infinite() {
  infinite(); // never returns — keeps pushing frames until the OS kills it
}
```

Most operating systems raise a segmentation fault or a dedicated stack overflow signal when this happens. Languages like Java and Python throw a `StackOverflowError` or `RecursionError`.

## Stack in Different Languages

- **C/C++**: Local variables and function parameters go on the stack by default. You must use `malloc` or `new` to allocate on the [heap](/programming/memory-management/heap).
- **Java / C#**: Primitive types (`int`, `bool`) and object *references* live on the stack; the objects themselves always live on the [heap](/programming/memory-management/heap).
- **Rust**: Stack allocation is the default. Heap allocation requires explicit types like `Box<T>`, `Vec<T>`, or `String`.
- **Python / JavaScript**: The runtime manages all allocation internally. Developers rarely interact with the stack directly, but it still exists and limits recursion depth.
- **Go**: Uses a *growable stack* — it starts small and is automatically expanded by the runtime as needed, making deep recursion safer.

## Stack vs Heap

The stack and [heap](/programming/memory-management/heap) are both memory regions but serve very different purposes:

- The stack is **fast, automatic, and scoped** — data lives only as long as the function that created it.
- The [heap](/programming/memory-management/heap) is **flexible and long-lived** — data can outlive any function, but requires explicit or garbage-collected management.

Use the stack for short-lived, fixed-size data. Use the [heap](/programming/memory-management/heap) when you need data to persist after the creating function returns, or when the size is unknown at compile time.
