# Canvas — Chains, Groups & Chords

Canvas is Celery's system for composing multiple tasks into workflows. Instead of writing orchestration logic in your application (which would require blocking and waiting), Canvas lets you declare *how* tasks should relate to each other and then hand the whole workflow to Celery to execute asynchronously.

## Why Orchestration Is Tricky

The naive approach to a multi-step pipeline looks like this:

```
result1 = step1.delay(data)
result2 = step2.delay(result1.get())   ← blocks here waiting
result3 = step3.delay(result2.get())   ← blocks here too
```

This works, but it's terrible:
- The application blocks on each `.get()`, defeating the purpose of async
- If this code runs inside a task, it holds a worker slot hostage while waiting
- It leads to the deadlock scenario described in the results lesson

Canvas solves this by letting you **describe the pipeline** before any task runs, and then executing it without blocking.

## The Core Idea: Callbacks

Celery's workflow system is built on callbacks. Instead of "run step 1, then step 2," you say "run step 1, and when it finishes, automatically enqueue step 2 with step 1's result."

No process is blocking. When step 1 finishes, the worker looks at its callback list, enqueues step 2, and moves on to other work. Step 2 runs when a worker is available.

```
Worker runs step1
  → step1 completes → result = 42
  → worker sees: "when done, enqueue step2 with 42"
  → worker enqueues step2(42)
  → worker moves on
Another worker (or same) runs step2(42)
```

## Chain — Sequential Steps

A chain is a linear pipeline where each task's output becomes the next task's input.

```
fetch → parse → store
  ↓result→  ↓result→
```

The chain is declared all at once. No task runs until you launch the chain. When you do, only the first task is enqueued immediately. Each subsequent task is a callback that fires after the previous one completes.

**Mental model:** A relay race. Each runner (task) gets the baton (result) from the previous runner and carries it forward to the next.

**Immutable steps:** Sometimes a task in the chain shouldn't receive the previous result — it's a side effect, not a data transformation. An immutable step ignores the incoming result and only uses the arguments you baked into it.

```
add → multiply → send_notification
  ↓result→  ↓result→  ✗ (notification doesn't need the number)
```

## Group — Parallel Execution

A group runs multiple tasks simultaneously. All tasks start at once. When all have finished, you get a list of all their results.

```
         ┌── fetch(url1) ──┐
launch ──┤── fetch(url2) ──├── all done → [result1, result2, result3]
         └── fetch(url3) ──┘
```

The group is ideal when you have independent work that doesn't depend on each other's results. Parallel execution reduces total time from (sum of all durations) to (longest single duration).

**Example:** fetching 10 URLs. Sequentially: 10 × 2s = 20s. In a group with 10 workers: ~2s total.

**Mental model:** Sending the same question to 10 experts simultaneously and collecting all their answers, rather than asking them one by one.

## Chord — Parallel + Callback

A chord is a group with a callback. All the parallel tasks run, and when the *last one finishes*, a final callback task runs with the collected results as input.

```
         ┌── process(item1) ──┐
launch ──┤── process(item2) ──├──► aggregate(all_results)
         └── process(item3) ──┘
```

This is the **map-reduce pattern**: map a function over many inputs in parallel, then reduce all results into one.

**How does Celery know when "all" have finished?** The result backend tracks it. Each task in the header group, as it completes, checks in with the backend. A counter decrements. When it hits zero, the callback is enqueued. This is why chords require a result backend.

**Mental model:** You send questionnaires to 100 customers (the group). When the last response comes in, you run a statistical analysis on all responses (the callback). You couldn't run the analysis until every questionnaire was returned.

## Combining Primitives

Chains, groups, and chords can be nested and combined to express complex workflows.

**Fan-out, then aggregate:**
```
fetch all URLs in parallel (group)
  → combine into one document (callback)
```

**Pipeline of parallel stages:**
```
Stage 1: download 100 files (group)
  → Stage 2: parse each file (chord callback that spawns new group)
  → Stage 3: merge all parsed data (final callback)
```

**Mixed:**
```
step1 → [step2a + step2b + step2c in parallel] → step3
  chain(step1, chord(group(...), step3))
```

The composition model is the same building blocks repeated at different scales.

## The Key Insight: No Blocking

The entire point of Canvas is that nothing blocks. Here's the contrast:

**Without Canvas:**
```
task1 runs → app waits → task2 runs → app waits → task3 runs
```

**With Canvas:**
```
app declares: task1 → task2 → task3
app enqueues and moves on (no waiting)
task1 runs on worker → triggers task2 → triggers task3
```

Worker slots are never held idle waiting for other tasks. The callback mechanism means each step is enqueued only when its dependencies are ready, and the enqueuing is done by Celery infrastructure, not by a blocking application thread.

## Signatures as the Building Block

Every Canvas primitive is built from **signatures** — serializable descriptions of task calls. A chain is a list of signatures. A group is a set of signatures. A chord is a group of signatures plus one callback signature.

Because signatures are just data (JSON), they can be passed around, stored, and inspected before execution. This is what makes it possible to describe the entire workflow upfront and hand it off to Celery.

## When to Use Canvas

**Use a chain** when steps must happen in order and each step depends on the previous result.

**Use a group** when steps are independent and can run in parallel.

**Use a chord** when you need to fan out work across many tasks and then aggregate the results.

**Don't use Canvas** for simple single-task operations — it adds unnecessary complexity. Canvas is for multi-step workflows where the relationship between tasks matters.
