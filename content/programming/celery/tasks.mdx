# Tasks & Signatures

A task is the fundamental unit of work in Celery. Understanding what a task really is — and how calling one differs from a normal function call — is central to using Celery correctly.

## What a Task Is

A task is a Python function that has been **registered** with Celery. Registering it gives Celery two things:

1. A **name** — a unique string identifier (usually the module path + function name). This name is what gets put into the broker message, not the actual code.
2. A **callable** that workers know to look up by that name when they receive a message.

The message that travels through the broker is just a small piece of JSON: the task name, the arguments, and some metadata. The actual code lives in the worker's Python environment.

```
Broker message:
{
  "task": "myapp.tasks.send_email",
  "args": ["alice@example.com"],
  "kwargs": {"subject": "Hello"},
  "id": "abc-123-uuid"
}
```

The worker sees this, looks up `myapp.tasks.send_email` in its registry, and calls it with those arguments.

## Three Ways to Call a Task

### 1. Calling it normally (synchronous)

You call the function like any Python function. It runs immediately in the current process. No broker involved. This is useful for testing.

### 2. `.delay()` — fire and forget

This sends the task to the broker and returns immediately. The actual execution happens in a worker, asynchronously. You get back a handle (an `AsyncResult`) but you don't have to use it.

The handle contains a task ID. If you care about the result, you can use this ID to retrieve it later.

### 3. `.apply_async()` — fire with options

Same as `.delay()` but gives you control over *when* the task runs, which queue it goes to, when it expires, and more.

You can say: "run this task in 5 minutes" or "put this task in the high-priority queue" or "discard this task if it hasn't started within an hour."

## The AsyncResult Handle

When you call a task asynchronously, you immediately get back a handle — think of it like a claim ticket at a coat check. You hand over your coat (the task), get a ticket (the AsyncResult), and can come back later to retrieve it.

The handle lets you ask:
- Has this task finished yet?
- Did it succeed or fail?
- What did it return?

You ask these questions by checking the **result backend** — the store where workers write their output. Without a result backend, the handle is effectively useless.

```
App → broker → worker runs task → writes result to backend
                                           ↑
App → uses handle → queries backend → gets result
```

## Task States

Every task goes through states:

```
PENDING → STARTED → SUCCESS
                  ↘ FAILURE
                  ↘ RETRY
```

**PENDING** — the task has been sent but no worker has started it yet (or the ID is unknown).

**STARTED** — a worker picked it up and is running it (only tracked if you enable this explicitly).

**SUCCESS** — completed without error. Result is stored.

**FAILURE** — raised an exception. The exception is stored.

**RETRY** — the task failed and is being re-queued to try again.

**REVOKED** — the task was cancelled before or during execution.

## Retries — The Logic

Tasks fail. Networks time out. External APIs go down. Retries let you handle transient failures gracefully without losing work.

The retry logic works like this:

1. Task runs and raises an exception.
2. Instead of failing permanently, the task re-enqueues itself with a delay.
3. After the delay, a worker picks it up and tries again.
4. This repeats up to a maximum number of attempts.
5. If all attempts fail, the task enters FAILURE state permanently.

**Exponential backoff** is the standard retry strategy: wait 1 second, then 2, then 4, then 8... This prevents hammering a struggling external service with rapid retries.

**Jitter** adds randomness to the delay. Without jitter, if 1000 tasks all fail at the same moment, they all retry at the same moment — creating a thundering herd. Jitter spreads them out.

```
Without jitter:  all 1000 tasks retry at t=30s → spike
With jitter:     tasks retry between t=25s and t=35s → spread
```

## Idempotency — The Most Important Design Principle

With retries and at-least-once delivery, a task may run more than once. This makes **idempotency** essential.

An idempotent task produces the same result whether it runs once or ten times.

**Non-idempotent (dangerous):**
```
Create a database row with this data
→ runs twice → two duplicate rows
```

**Idempotent (safe):**
```
Create a database row IF one doesn't already exist with this key
→ runs twice → still one row
```

The pattern is always: check before acting, or use upserts, or design the operation so repetition has no harmful side effect.

## Signatures — Describing a Call Without Making It

A **signature** is a serializable description of a task call. It captures:
- Which task to call
- What arguments to pass

But it doesn't execute the task immediately. It's a recipe you can pass around, store, or execute later.

Think of it like the difference between *eating a sandwich* and *writing down the recipe for a sandwich*. A signature is the recipe.

Signatures become essential for workflow composition — you build a pipeline by connecting signatures together, and Celery executes them in the right order with results flowing between steps.

There are two flavors:

**Mutable signature** — when this task runs as part of a pipeline, it receives the result of the previous step as its first argument.

**Immutable signature** — this task ignores whatever the previous step returned. It only uses the arguments baked into the signature.

This distinction matters in chained workflows: sometimes you want to pass data forward, sometimes you want an independent side effect (like sending a notification) that doesn't depend on the previous result.

## Revoking Tasks

You can cancel a task that's waiting in the queue. If it hasn't been picked up yet, it won't run.

If a task is already running, you can send a termination signal to the worker running it. This is a more aggressive operation and the worker may not respond immediately.

Revocation is useful for user-initiated cancellations: "I submitted a report generation but I changed my mind."

## Soft and Hard Time Limits

Tasks can have time limits — to prevent runaway tasks from blocking a worker slot indefinitely.

**Soft limit** — after N seconds, the task receives a signal (an exception). It can catch this, clean up, and exit gracefully.

**Hard limit** — after N seconds, the worker kills the task process unconditionally. No cleanup possible.

The typical pattern: set the soft limit slightly lower than the hard limit. Give the task a chance to clean up on soft, then force-kill on hard.
