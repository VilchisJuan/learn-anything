# Celery Worker

A worker is a process that watches the broker, picks up tasks, and runs them. It's the engine room of Celery — the place where your actual code executes.

## What a Worker Does, Step by Step

1. It connects to the broker on startup.
2. It tells the broker: *"I'm ready. Send me tasks from queue X."*
3. It waits. When a task message arrives, it deserializes it (unpacks the function name and arguments).
4. It executes the function.
5. If a result backend is configured, it stores the return value.
6. It acknowledges the message (tells the broker: *"done, remove this from the queue"*).
7. It goes back to waiting.

This loop runs continuously. Workers are meant to be long-lived processes.

## Concurrency: Doing Multiple Things at Once

A single worker process can handle more than one task at a time. It achieves this through a **pool** — a set of sub-processes or threads working in parallel.

Think of the worker as a manager and the pool as a team of workers the manager oversees:

```
Worker (manager)
  ├── Pool slot 1 → task A (running)
  ├── Pool slot 2 → task B (running)
  ├── Pool slot 3 → task C (running)
  └── Pool slot 4 → (idle, waiting)
```

The number of slots is the **concurrency** setting. By default it equals the number of CPU cores on the machine.

## The Two Types of Work — and Why It Matters

Not all tasks are the same. There are two fundamentally different kinds of slow operations:

**CPU-bound work** — the processor is actually busy computing. Examples: image encoding, number crunching, machine learning inference. The CPU is the bottleneck.

**I/O-bound work** — the processor is mostly waiting. Examples: fetching a URL, querying a database, reading a file. The network or disk is the bottleneck, not the CPU.

This distinction matters because it determines which pool strategy is optimal.

## Pool Strategies

### Prefork (Multiple Processes)

The default. Spawns N independent Python processes. Each process runs tasks in isolation.

```
Worker
  ├── Process 1  (full Python runtime, own memory)
  ├── Process 2  (full Python runtime, own memory)
  └── Process 3  (full Python runtime, own memory)
```

Good for CPU-bound tasks because each process runs on its own CPU core in true parallel. No Python GIL interference.

Drawback: each process has memory overhead (~50-100MB per process). With 16 processes, that's 800MB just for the pool.

### Gevent / Eventlet (Green Threads)

Instead of real OS processes or threads, these use **green threads** — cooperative micro-threads managed by a library, not the OS.

The key insight: when a green thread hits an I/O wait (a network call, a sleep), it voluntarily pauses and lets another green thread run. This means one OS thread can juggle thousands of concurrent I/O operations.

```
Single OS Thread
  → task A starts HTTP call → pauses
  → task B starts DB query → pauses
  → task C completes → yields back
  → task A resumes when HTTP responds
```

Good for I/O-bound tasks. You can set concurrency to 500 or 1000 without spawning 500 processes.

Useless for CPU-bound tasks — green threads don't provide real parallelism.

### Threads

Uses actual OS threads. The Python GIL limits true parallelism for CPU work, but threads are fine for I/O-bound tasks and have lower setup overhead than processes.

### Solo

No pool. Tasks run sequentially in the main process. Useful only for debugging.

## Task Prefetching

Workers don't wait passively for one task at a time. They **prefetch** — they pull several tasks from the broker before finishing their current ones. This reduces the round-trip latency of fetching each task individually.

The **prefetch multiplier** controls how many tasks a worker reserves per pool slot. A multiplier of 4 with 4 pool slots means the worker grabs up to 16 tasks at once.

This is efficient for fast, uniform tasks. But for slow or uneven tasks, prefetching causes a problem:

```
Slot 1: running a 10-minute task
Slot 2: running a 10-minute task
Reserved (prefetched): 8 more tasks sitting idle

→ Other workers have nothing to do even though the queue has work
```

For long or variable-duration tasks, set the multiplier to 1. Each slot only grabs one task at a time, ensuring fair distribution.

## Task Acknowledgment

When a worker picks up a task, when does it tell the broker *"I've handled this, you can delete it"*?

**Acknowledge on receipt (default):** The task is removed from the broker queue the moment the worker receives it — before execution even starts. If the worker crashes during execution, the task is gone.

**Acknowledge on completion (late ack):** The task stays in the broker queue until execution finishes. If the worker crashes, the broker re-delivers the task to another worker.

Late acknowledgment makes your system more resilient, but it means tasks might run more than once. That's why tasks should be designed to be **idempotent** — running them twice produces the same outcome as running them once.

## Multiple Workers

You can run many workers, all connected to the same broker. They compete to pick tasks from the queue — whoever is free and fast enough picks the next task.

```
Broker Queue: [task1] [task2] [task3] [task4]
                  ↙          ↘
           Worker A        Worker B
           (server 1)      (server 2)
```

This is the horizontal scaling model. Adding workers increases throughput without changing application code.

## Queues and Specialization

By default all tasks go into one queue. But you can have multiple queues and tell different workers to listen to different ones.

Why? Because a heavy, slow task (video encoding) and a lightweight task (send email) shouldn't compete for the same pool slots. If 4 video-encoding tasks fill all slots, email sending is stuck.

With specialized queues:

```
"emails" queue  ──► email-worker  (20 slots, gevent)
"heavy" queue   ──► heavy-worker  (2 slots, prefork)
"default" queue ──► general-worker (8 slots, prefork)
```

Each queue gets the right concurrency model and the right amount of resources.

## Worker Lifecycle

```
Start → Connect to broker → Announce readiness
      → Fetch task → Execute → Store result → Acknowledge
      → Repeat until shutdown signal

Shutdown (graceful): finish current tasks, then exit
Shutdown (forced):   exit immediately, in-flight tasks may be re-queued
```

Workers should be managed as services (systemd, Docker, supervisor) so they restart automatically if they crash.
