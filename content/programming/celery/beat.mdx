# Celery Beat

Beat is Celery's scheduler. Its job is simple: at the right moment, it puts a task into the broker queue so a worker can pick it up.

Beat does not execute tasks. It only schedules them.

## The Alarm Clock Analogy

Think of Beat as an alarm clock, and the broker as a to-do board.

When the alarm goes off, Beat writes a new task on the to-do board. A worker sees the task, picks it up, and does the work. Beat's job ends there — it goes back to watching the clock.

```
Beat (alarm clock)
  → "It's Monday 9 AM"
  → writes task on broker ──► Worker picks it up and runs it
  → goes back to watching the clock
```

Beat runs continuously as a separate process. It wakes up frequently (roughly every second) to check whether any scheduled task is due.

## How Beat Knows What to Run and When

Beat holds a **schedule** — a list of tasks and their timing rules.

Each entry says:
- Which task to call
- How often (or at what time)
- What arguments to pass

The schedule can be written as code in your configuration, or stored in a database and edited dynamically.

## Types of Timing Rules

**Fixed interval** — run every N seconds. Simple. No cron syntax needed.

```
Every 30 seconds → heartbeat task
Every 1 hour     → sync cache task
```

**Crontab** — run at specific times, like Unix cron. Expressive: you specify the minute, hour, day of week, day of month, and month.

```
Every Monday at 9 AM
Every day at midnight
Every 15 minutes
The 1st of every month at 6 AM
```

**Solar** — run at sunrise or sunset for a given geographic location. Unusual, but built-in.

## The Schedule File

Beat needs to remember when each task last ran, so it doesn't re-fire tasks if it restarts.

It stores this in a local file (a small database). On startup, it reads this file to know where it left off. If the file is missing or corrupted, Beat thinks no tasks have ever run and may fire them all immediately.

This is an important operational detail: **protect the schedule file**. Don't delete it. Don't let it live in ephemeral storage (like a Docker container layer that gets wiped on restart).

## Dynamic Schedules

The built-in schedule is static — defined in code. To change it, you modify the code and restart Beat.

But sometimes you need a schedule that changes at runtime — think a multi-tenant SaaS where each customer sets their own report timing. For this, the schedule can live in a database. Beat reads from the database on each tick, so changes take effect without any restart.

This is what `django-celery-beat` provides: a database-backed schedule with an admin interface. You can add, edit, and delete periodic tasks through a UI without touching code.

## The Critical Rule: One Beat Process

Beat is not safe to run in multiple instances. If two Beat processes are running against the same schedule, both will fire each task — every task runs twice.

```
Beat-1: "It's 9 AM Monday, enqueue weekly-report"
Beat-2: "It's 9 AM Monday, enqueue weekly-report"
→ weekly-report runs TWICE
```

Always run exactly one Beat process. In a multi-server deployment, this means Beat must be a singleton service, not something that runs on every server.

## Beat vs. Cron

Beat is often compared to Unix cron. They're conceptually similar, but Beat has advantages in a Celery context:

| | Unix Cron | Celery Beat |
|---|---|---|
| Where tasks run | On the cron machine | On any available worker |
| Scale | Single machine | Distributed workers |
| Task history / retries | No | Yes (via Celery) |
| Dynamic scheduling | No (edit crontab) | Yes (database scheduler) |
| Integration with app | Manual scripts | Native Celery tasks |

The key difference: cron runs a command on the cron machine. Beat *enqueues* a task that a worker runs. The work is distributed.

## Timezone Awareness

Beat respects timezone settings. A crontab that says "9 AM" means 9 AM in whatever timezone you configure. If you don't configure one, it defaults to UTC — which may not be what you intend.

Always set the timezone explicitly. Daylight saving time transitions can cause tasks to fire at unexpected times if the timezone is misconfigured.

## Beat and Workers Together

Beat and workers are separate processes. In development, they can run together in a single process for convenience. In production, always keep them separate — a crash in the worker process should not stop the scheduler, and vice versa.

```
Production setup:
  [Beat process]    ──► Broker ──► [Worker process(es)]

Development shortcut:
  [Beat + Worker in one process]   (never in production)
```

## What Happens If Beat Goes Down?

If Beat stops, no new scheduled tasks are enqueued. Workers keep running their current tasks, but no new scheduled work arrives.

When Beat restarts, it checks the schedule file. If it knows a task was due while it was down, it may fire it immediately upon restart (depending on configuration). This is called **catchup** — and it can cause a burst of tasks all at once. Consider whether catchup is desirable for your use case, and disable it if not.
