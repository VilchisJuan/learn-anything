# Result Backends

A result backend is where workers write their output — the return value and final state of a completed task. Your application can then query this store to find out what happened.

## Do You Actually Need One?

The first question to ask is whether you need a result backend at all.

Many tasks are **fire-and-forget**: send this email, log this event, clean up these old records. You don't need to know what they returned. The side effect is the point.

For these tasks, a result backend is waste — extra writes, extra storage, extra complexity. You can disable result storage for these tasks entirely.

A result backend becomes necessary when:
- You need to retrieve a task's return value
- You need to check whether a specific task succeeded or failed
- You're using workflows that chain tasks together (chords specifically require it internally)
- You want to display task status to the user ("Your export is 60% done")

## How It Works

The flow is simple:

```
Worker finishes task
  → writes {id: "abc", status: "SUCCESS", result: 42} to backend
  → worker moves on

App queries backend with id "abc"
  → gets {status: "SUCCESS", result: 42}
  → uses the result
```

The backend is just a key-value store. The key is the task ID. The value is a JSON blob with the status and return value.

## Task States in the Backend

When you query a task ID, you get one of these states:

**PENDING** — the backend has no record of this task. It either hasn't run yet, or result storage is disabled, or the ID is wrong.

**SUCCESS** — the task completed without error. The return value is stored.

**FAILURE** — the task raised an exception. The exception details are stored.

**RETRY** — the task failed and is being retried.

**REVOKED** — the task was cancelled.

Note that PENDING doesn't mean "currently running." It means the backend doesn't know about it. A task might be running right now, but until it finishes and writes to the backend, the backend sees it as PENDING.

## Result Expiration

Results don't need to live forever. Once your app has read the result, the stored record has no value. Results accumulate over time — a busy system can generate thousands of result records per hour.

Set an expiration time. After that window, results are automatically deleted. The right window depends on your use case: if your app checks results within minutes, a few hours is plenty. If results need to be queryable days later, set accordingly.

## Backends and Their Trade-offs

### Redis

The most common backend. Fast, simple, and often already in your stack as the broker too.

Results are stored as Redis keys with an automatic expiration (TTL). Redis handles cleanup automatically. No manual maintenance needed.

**Trade-off:** Redis is in-memory. If you need results to survive a Redis restart, you must enable persistence. Results are not queryable — you can only look them up by task ID, not by task name or date range.

### Database (SQL)

Results are stored as rows in a SQL table. This enables queries: "show me all failed tasks from the last 24 hours," "how many tasks of type X ran this week?" You get full SQL expressiveness.

**Trade-off:** Every task completion requires a database write. Under high throughput, this adds significant load to your database. The result table can grow very large without careful cleanup.

Good fit when you need an audit trail or want to surface task history in an admin interface.

### Cache (Django Cache Framework)

Uses whatever caching backend Django is configured with. If that's Redis, you get Redis-backed results without using the Redis broker directly — useful for separation of concerns.

## The Claim Ticket Model

Think of the result backend like a parcel pickup counter.

When you drop off a task (ship a parcel), you get a ticket with a tracking number. You go home. Later, you return to the counter, hand over your ticket, and pick up your parcel (the result).

The counter (backend) is the intermediary. It holds your parcel until you claim it. If you never come back, the parcel eventually gets disposed of (result expiration).

```
App → submit task → gets ticket (task ID)
Worker → completes task → drops result at counter (backend)
App → shows ticket → picks up result
```

## Querying Results Across Multiple Tasks

When you run multiple tasks in parallel (a group), you get a handle that tracks all of them. You can ask:

- Are all of them done?
- What did each one return?
- Did any fail?

The backend stores each task individually. The group handle is just a collection of task IDs. Celery queries each one and assembles the final picture.

## The Danger of Blocking on Results Inside Tasks

A common mistake: a task calls another task and then waits for its result. This looks natural but can deadlock.

Imagine a worker with 4 slots, all occupied:

```
Slot 1: Task A → spawns Task B → waits for Task B's result
Slot 2: Task A → spawns Task B → waits for Task B's result
Slot 3: Task A → spawns Task B → waits for Task B's result
Slot 4: Task A → spawns Task B → waits for Task B's result
```

Task B is waiting in the queue. But all 4 slots are blocked waiting for Task B. Task B can never start. Deadlock.

The fix: use Celery's Canvas primitives (chains and chords) to express result-dependent workflows without any task blocking on another.
