# Flower — Monitoring

Flower is a web dashboard that gives you a live view into what your Celery cluster is doing. It answers the questions you can't answer by looking at logs alone: Are my workers healthy? How deep is the queue? Why did that task fail?

## The Problem Without Monitoring

Without visibility, a Celery cluster is a black box. Tasks go in, results come out (sometimes), and when things go wrong you have no idea why.

Common invisible failures:
- A worker crashed and no one noticed — tasks are queuing up silently
- Beat stopped running — scheduled jobs are simply not happening
- A task is stuck in RETRY loop, burning retries silently
- The queue is growing because workers can't keep up
- One worker is handling 90% of the work because others are misconfigured

Flower makes all of these visible at a glance.

## What Flower Shows You

### Workers

The workers view is the first place to check when something seems wrong.

You see every worker that has ever connected, its current status (online or offline), how many tasks it's currently running, and how many it has processed in total.

If a worker disappears from this list (or turns offline), it crashed. If a worker shows zero active tasks when you know the queue is full, its pool is misconfigured or stuck.

### Tasks

The tasks view is a searchable, filterable history of every task execution. For each task you can see:

- Its state (SUCCESS, FAILURE, RETRY…)
- Which worker ran it
- When it started and how long it took
- What arguments it was called with
- What it returned, or what exception it raised with the full traceback

This is invaluable for debugging failures. Instead of hunting through log files across multiple machines, you go to the tasks view, filter by FAILURE, and click the task to see exactly what went wrong.

### Queues (Broker Tab)

The broker tab shows you the **depth** of each queue — how many tasks are waiting to be picked up.

This is the most important operational metric in a Celery system:

```
Queue depth = 0          → healthy, workers are keeping up
Queue depth slowly grows → you need more workers
Queue depth = 0 always   → you might have too many idle workers
Queue depth spikes       → a sudden burst of tasks (normal or a bug?)
```

A queue that grows without bound means your system is falling behind. Tasks will eventually be delayed so long they become useless.

### Monitor

Live charts showing throughput over time: how many tasks per second are succeeding or failing. Lets you spot anomalies — a sudden drop in throughput might mean workers went offline; a spike in failures might mean an upstream service went down.

## Key Operations from Flower

**Revoking a task** — cancel a task that's still waiting in the queue, or terminate one that's currently running. Useful when a bug causes tasks to pile up and you need to stop the bleeding without restarting workers.

**Retrying a failed task** — re-enqueue a specific failed task with the same arguments. Useful for one-off failures caused by transient issues (network blip, API rate limit) that have since resolved.

**Inspecting a worker** — see exactly what tasks a specific worker is running right now, what's reserved (prefetched), and what's registered. Helps diagnose why a specific worker seems slow or stuck.

**Shutting down a worker** — gracefully terminate a worker from the UI, without SSH access to the machine.

## Why Flower Is a Separate Process

Flower connects directly to the broker and the result backend. It listens to Celery's event stream — workers emit events as they pick up tasks, complete them, or fail. Flower aggregates these events in real time.

Because it's separate from both workers and your application, it can run independently. A worker crash doesn't take Flower down. A Flower crash doesn't affect your workers.

This separation is intentional: monitoring should be decoupled from the thing being monitored.

## Authentication Is Not Optional in Production

Flower has no authentication by default. Anyone who can reach the port can see task arguments, return values, exception details, and control workers.

Task arguments often contain sensitive data: user IDs, email addresses, API tokens, partial PII. A publicly accessible Flower instance is a data leak.

In production, always:
- Put Flower behind a reverse proxy (Nginx, Caddy)
- Require authentication (basic auth at minimum)
- Restrict access by IP or VPN
- Use HTTPS

## Prometheus Integration

Flower exposes a `/metrics` endpoint in Prometheus format. This lets you pull Celery metrics into your existing monitoring stack (Prometheus + Grafana) and set up alerts.

Useful alerts to configure:
- Worker count drops below a threshold
- Queue depth exceeds a threshold for more than N minutes
- Task failure rate exceeds N% over M minutes
- No tasks processed in the last N minutes (Beat might be down)

The dashboard is great for human inspection. Alerts are what actually wake you up at 3 AM when something breaks.

## When Flower Isn't Enough

Flower is a real-time view — it's not a long-term analytics store. It holds a limited history of task events in memory (or in a local SQLite file if you enable persistence).

For long-term trend analysis, task SLA tracking, or correlating Celery failures with other system events, you need to ship metrics and logs to a proper observability platform (Datadog, Grafana, Elastic, etc.).

Think of Flower as the cockpit instrument panel — essential for flying the plane right now, but not a substitute for the flight data recorder.
