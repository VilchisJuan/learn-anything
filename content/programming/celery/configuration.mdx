# Configuration & Best Practices

Getting Celery running locally takes minutes. Getting it right in production requires deliberate choices about reliability, security, and resource management. This lesson explains the *why* behind those choices.

## Serialization: Why JSON and Not Pickle

When a task is enqueued, its arguments must be serialized into a format that can travel through the broker. Celery supports several formats, but the choice matters enormously for security.

**Pickle** is Python's native serialization format. It can represent almost any Python object. It's convenient — you can pass complex objects as task arguments without thinking about it.

The problem: **pickle is executable**. When a worker deserializes a pickle payload, it executes Python code embedded in the payload. If anyone can write to your broker (or intercept messages), they can execute arbitrary code on your workers.

**JSON** can only represent simple data types — strings, numbers, lists, dictionaries. It cannot execute code. A malicious JSON payload is just malformed data, not a code injection.

The rule: always use JSON. If you need to pass complex objects, serialize them to simple types first (pass an ID and look up the object inside the task, rather than passing the object itself).

## Acknowledgment and Idempotency — The Reliability Contract

There is a fundamental tension in distributed systems between **performance** and **reliability**:

- Acknowledge immediately → faster, but tasks can be lost if the worker crashes during execution
- Acknowledge after completion (late ack) → slower, but tasks survive worker crashes

Late acknowledgment gives you the stronger guarantee: work is never silently lost. But it introduces a new requirement: tasks must be idempotent. Because with late ack, a task *will* run more than once in failure scenarios.

These two settings go together. Enabling late ack without designing idempotent tasks just trades one problem (data loss) for another (duplicate processing).

The prefetch multiplier should also be set to 1 alongside late ack. With late ack, a task stays in the "unacknowledged" state on the broker until it finishes. If a worker has prefetched many tasks and crashes, all those prefetched tasks are re-delivered. Prefetching one at a time limits the blast radius.

## Time Limits — Preventing Runaway Tasks

Without time limits, a task that gets stuck (infinite loop, deadlocked external call, network hung) holds a worker slot forever. With 8 slots and 8 stuck tasks, your entire worker is frozen.

Time limits are the circuit breaker:

**Soft limit** — sends an exception to the running task after N seconds. The task can catch this, log the timeout, clean up open resources, and exit cleanly.

**Hard limit** — kills the task process after N seconds, unconditionally. No cleanup possible.

The right pattern is: soft limit slightly lower than the hard limit. The soft limit gives the task a window to handle the timeout gracefully. The hard limit is the guarantee that the slot will eventually be freed no matter what.

Every task should have a time limit. The appropriate value depends on the task — an email send might have a 30-second limit, while a video encoding task might have a 2-hour limit.

## Queue Design — Isolation by Work Type

All tasks sharing one queue creates a problem called **head-of-line blocking**: a slow task blocks all the fast tasks behind it.

```
Queue: [10-min video encode][email][email][email][email]
All 4 worker slots busy encoding → emails wait 10 minutes
```

Separate queues with dedicated worker pools solve this:

```
"heavy" queue  → 2 workers (prefork, for CPU work)
"emails" queue → 20 workers (gevent, for I/O work)
"default" queue → 8 workers (general use)
```

Fast tasks never wait behind slow ones. Each queue can use the concurrency model that fits its workload. Each can be scaled independently.

As a rule of thumb: if two types of tasks have dramatically different durations, resource requirements, or failure modes, they should be in separate queues.

## Visibility Timeout — A Redis-Specific Gotcha

When using Redis as a broker, there's a hidden footgun: the **visibility timeout**.

When a worker picks up a task, Redis starts a timer. If the task isn't acknowledged before the timer expires, Redis assumes the worker died and re-delivers the task to another worker.

If your task runs longer than the visibility timeout, it gets re-delivered — and now two workers are running the same task simultaneously.

The fix is simple: set the visibility timeout higher than the longest task you ever expect to run. If your slowest task takes 4 hours, set the timeout to 5+ hours.

## Result Expiration — Avoiding Unbounded Growth

Every completed task writes a record to the result backend. Without expiration, these records accumulate indefinitely.

For a system processing 1000 tasks per hour, that's 24,000 records per day, 720,000 per month — all just sitting there, most of them never queried again after the first few seconds.

Set an expiration window that reflects your actual query patterns. If your app checks task status within the first minute, a few hours of retention is generous. Results older than your expiration window are automatically cleaned up.

For tasks where you never check the result, disable result storage entirely. This is both simpler and more efficient.

## Broker Connection Resilience

Workers maintain a persistent connection to the broker. When the broker restarts (routine maintenance, crash, failover), that connection breaks. Workers need to know how to reconnect automatically.

Without explicit reconnection configuration, a broker restart might leave workers in a broken state — connected but not receiving tasks, silently idle.

Configure workers to retry connection on startup and to reconnect automatically when the connection drops. Set a heartbeat interval so dead connections are detected quickly rather than hanging silently.

## Deployment: Separate Processes, Managed by a Service Manager

Three things must always run as separate processes:

1. **Workers** — one or more, depending on load
2. **Beat** — exactly one
3. **Flower** — one (optional but recommended)

Each should be managed by a service manager (systemd, supervisord, or Docker with a restart policy). If any of them crash, the service manager restarts them automatically. Without this, a crash at 3 AM goes unnoticed until someone checks in the morning.

Beat is especially critical: it's a singleton that must always be running. A Beat outage means all scheduled tasks silently stop. Most outages of this type go unnoticed for hours because there's no visible error — things just don't happen.

## Worker Memory Management

Python processes accumulate memory over time — libraries cache things, garbage collection isn't perfect, and long-running processes can develop slow memory leaks.

Workers can be configured to recycle themselves after processing N tasks. This limits memory growth: after 1000 tasks, the worker process exits and the service manager spawns a fresh one. The queue isn't lost — a new worker picks up the next task immediately.

This is a pragmatic solution that trades a tiny bit of throughput (the restart cost) for predictable memory usage over long periods.

## The Production Checklist (Mental Model)

When evaluating a Celery deployment, ask these questions:

**Reliability:** If a worker crashes mid-task, is the work re-queued or lost? (late ack)

**Safety:** If a task is re-queued and runs twice, does that cause data corruption? (idempotency)

**Security:** Can arbitrary code be injected through task messages? (serializer)

**Responsiveness:** Can a slow task type starve fast task types? (queue isolation)

**Correctness:** Can a long task be re-delivered because of an undersized timeout? (visibility timeout)

**Efficiency:** Is the result backend filling up with records nobody reads? (result expiration)

**Operability:** If any component crashes, will it restart automatically? (service management)

**Visibility:** Will you know when something goes wrong before users complain? (monitoring + alerts)

Each of these is an independent concern. A system that gets all of them right is robust. Missing any one of them is a latent failure waiting to surface under the right conditions.
