# Brokers

The broker is the middleman between your application and your workers. It holds task messages until a worker is ready to process them. Without a broker, there's no way to pass work between processes.

## What a Broker Actually Does

A broker is fundamentally a **durable queue**. Your application writes to one end. Workers read from the other end.

```
App writes ──► [task][task][task] ──► Worker reads
               (queue / broker)
```

But it's not just a simple pipe. A broker also handles:

- **Durability** — if the broker restarts, messages can survive (depending on configuration)
- **Multiple queues** — tasks can be categorized and routed to different worker pools
- **Delivery guarantees** — ensuring tasks aren't silently dropped
- **Multiple consumers** — many workers can read from the same queue simultaneously

## The Two Main Brokers

### Redis

Redis is an in-memory data store. Celery uses its **list data structure** as a queue: the app pushes to one end, workers pop from the other.

Redis is the most common choice for Celery because:
- It's simple to set up
- It's very fast (everything in memory)
- It often already exists in a stack (for caching, sessions, etc.)
- It doubles as a result backend

**The trade-off:** Redis's queue mechanism is relatively basic. Messages live in memory first, so if Redis crashes without persistence configured, messages are lost. Acknowledgment is handled by Celery in software rather than by the broker protocol itself.

**Persistence:** Redis can write to disk in the background (snapshots) or log every write (append-only file). With append-only mode enabled, messages survive a Redis restart. Without it, messages are at risk.

**Analogy:** Redis as a broker is like a whiteboard. Fast, everyone can see it, but if someone erases it (crashes), everything is gone — unless you've been writing it down somewhere else too.

### RabbitMQ

RabbitMQ is a purpose-built message broker implementing the AMQP protocol. It was designed specifically to reliably route messages between processes.

**Key difference in how messages travel:**

In Redis, messages go directly into a queue (list). In RabbitMQ, messages first go to an **exchange**, which then routes them to one or more queues based on rules called **bindings**.

```
Redis:      App → Queue → Worker

RabbitMQ:   App → Exchange → [routing rules] → Queue → Worker
```

This extra layer of indirection enables more sophisticated routing — one message can go to multiple queues, or messages can be filtered before reaching a queue.

**Native acknowledgment:** RabbitMQ handles acknowledgment at the protocol level. When a worker picks up a message, RabbitMQ keeps it in a "unacknowledged" state. If the worker dies before acknowledging, RabbitMQ automatically re-delivers to another worker. This is guaranteed by the protocol, not by Celery code.

**Dead-letter queues:** RabbitMQ can route failed or expired messages to a separate "dead letter" queue automatically. This gives you a place to inspect messages that couldn't be processed, without losing them.

**Analogy:** RabbitMQ is like a postal system. Every package has explicit tracking, guaranteed delivery, and a process for returning undeliverable packages — but the system is more complex to set up and operate.

## How Workers and Brokers Interact

Workers maintain a **persistent connection** to the broker. They don't reconnect for every task — that would be too slow. Instead, they keep a channel open and listen continuously.

When a task arrives:
1. The broker pushes a notification over the existing connection
2. The worker receives the message
3. The worker may pre-fetch additional messages (to reduce round-trips)
4. The worker processes and acknowledges

This long-lived connection means the broker needs to detect dead connections (via heartbeats) and re-deliver their unacknowledged tasks.

## Visibility Timeout (Redis-specific concept)

With Redis as a broker, Celery implements a visibility timeout. When a worker picks up a task, it sets a timer. If the task isn't completed and acknowledged before the timer expires, Celery assumes the worker is dead and re-delivers the task.

This means: if your task takes longer than the visibility timeout, it will be re-delivered and run twice — even if the original worker is still running fine.

**The fix:** set the visibility timeout to be longer than your longest-running task.

## Queue Depth — The Key Health Metric

The most important thing to watch in a broker is **queue depth** — how many tasks are waiting.

```
Queue depth = 0        → workers are keeping up (or idle)
Queue depth growing    → workers can't keep up → add more workers
Queue depth = constant → balanced (tasks in ≈ tasks out)
```

A growing queue depth is the first sign you need to scale your workers. If the queue grows without bound, eventually tasks will be delayed so long they become useless.

## Choosing Between Redis and RabbitMQ

**Start with Redis** if:
- You already use Redis for anything else
- You want the simplest possible setup
- You don't need advanced routing or broker-level dead-letter queues
- Your reliability requirements are met by Celery's software-level retries

**Use RabbitMQ** if:
- You need guaranteed, protocol-level message delivery
- You need dead-letter queues at the broker level
- You need complex routing (one message to multiple queues)
- You want message priorities
- Your team already operates RabbitMQ

For most applications, Redis is the right starting point. RabbitMQ becomes worth its additional operational complexity when you need its specific features.
