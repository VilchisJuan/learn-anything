**Regular expressions** (regex or regexp) are sequences of characters that define a search pattern. They allow you to search, match, extract, and replace text with extraordinary precision — in a single line of code that would otherwise require dozens of lines of string manipulation logic.

Regex is not a programming language. It is a **pattern language** that runs inside another language. Nearly every programming language — Python, JavaScript, Go, Java, Ruby, Rust — has built-in regex support, and the patterns are largely the same across all of them.

## A Taste of What Regex Can Do

Without regex, checking whether a string is a valid email address requires loops, character comparisons, and flag variables. With regex:

```js
const emailRegex = /^[\w.+-]+@[\w-]+\.[a-zA-Z]{2,}$/;
emailRegex.test("user@example.com"); // true
emailRegex.test("not-an-email");     // false
```

Or find all hashtags in a tweet:

```js
const tags = tweet.match(/#\w+/g); // ["#regex", "#programming"]
```

Or reformat a date from `MM/DD/YYYY` to `YYYY-MM-DD`:

```js
"03/15/2024".replace(/(\d{2})\/(\d{2})\/(\d{4})/, "$3-$1-$2");
// → "2024-03-15"
```

## A Brief History

Regex has roots in theoretical computer science. In 1951, mathematician **Stephen Kleene** formalized the concept of regular languages and the notation to describe them (giving rise to the `*` operator, still called a "Kleene star"). In the 1960s, Ken Thompson implemented regex matching in the QED text editor, and later in the Unix tool `grep` — **G**lobally search for a **R**egular **E**xpression and **P**rint.

The POSIX standard in the 1980s, and then Perl's extended regex in the 1990s, added features like lookaheads, named groups, and backreferences. Today, most languages follow the **PCRE** (Perl-Compatible Regular Expressions) standard.

| Lexers & compilers | Tokenizing source code |
| Data extraction | Pulling numbers, names, URLs from text |
| URL routing | Matching URL patterns in web frameworks |

## How Regex Works (Conceptually)

A regex engine reads your pattern and your input string, then tries to find positions in the string where the pattern matches. The engine uses one of two strategies:

- **NFA (Non-deterministic Finite Automaton)** — used by most modern engines (PCRE, Python `re`, JavaScript). It tries each alternative and backtracks if it fails. Supports lookaheads and backreferences.
- **DFA (Deterministic Finite Automaton)** — used by tools like `grep -E`. Faster but cannot support features like backreferences.

You don't need to understand the theory to use regex effectively, but knowing that backtracking exists explains why some patterns can be slow.
